FlashBack

Goal
  • Build a fully functional web application for vocabulary training, that enables the user to
    • Set up an account, signing in with facebook, google or email
    • Create individual cards, fill in the word, translation and upload a picture from a folder or through a googlesearch
    • Group cards in decks
    • Go through decks and fill out cards, then get the result (right or wrong) and be shown the right answer
    • See their progress through a deck
    • Reset a deck and start again
  • Learn to work in a collaborative and asynchronous fashion in a team of UX Designer and Software Engineer.
Approach
  • Set up guidelines for our team
    • How do we want to deal with conflicts and how can we prevent them?
    • What communication channels do we want to use?
    • What is our goal - for the application and personally?
    • How do we keep up to date with our progress?
    • What deadlines do we set outselves?
  • Set up a Trello Board, a Slack Channel, and Google Docs
  • Create a new Project with Ruby on Rails on GitLab
  • Draw out a first Roadmap and Schema
  • Formulate questions frequently
  • Set up calls regularily
  • Reach out to colleagues and ask if they are willing to help with design and code review
  • What would be the most complex app that we would want to build with all features?
  • What is the MVP?
Trello Instructions:

  • Facilitator to track all tasks, their assigned developer, and their progress. The tasks sections are grouped together by color labels.
  • The Overview column holds all the main groups of tasks, for example the groupings of MVC - model, view and controller - styling, pop-ups and others. These cards are never moved, but only to be consulted and updated, ticking off the individual tasks located in the checklist.
  • According to these checklists, each item is placed on an individual task card with the same color label and a developer assigned to it.
  • Task cards are accompanied by a feature branch - ideally named the same or similar - and moved to the column correspondent to their progress: To Do, Started, Review, and Done.
  • A card that reaches the Review column, has been pushed to GitHub and a pull request with detailed accompanying message has been issued.
  • The assigned team member sends a notification to the team on slack, so the other team member can review the code in the pull request, make suggestions or request changes.
  • Merges will be accomplished by the member not uploading the code.
  • The team is informed of the merge, so they can pull the updated master branch onto their local repositories.

  • Vue.js, RoR as an API, and TDD

    After doing a deep dive into Vue.js (and absolutely loving it!) and learning more about using Ruby on Rails as an API, I started a completely new repository, leaving the first version of the flashcard app behind, that I started building using only RoR. This gave me a chance to rethink the application more and apply all the newly acquired knowledge.

    I am thrilled about how the project is coming along and learning so much along the way. I want to put an emphasis on the territory of TDD, which I didn't know a lot about and was somehow a little intimidating by. However, learning from a colleague Juan Vqz and his tutorial on Vue and RoR that all changed rapidly, and I am all aboard, seeing how much easier it makes life in the long run. So thank you for that, Juan!

  • Implementing an API to fetch pictures from Unsplash

    Inside this, limiting the amount of pictures displayed with the option to show more results. The amount of learning you can achieve by only one “little" feature is immense and very satisfying. I had been working on this feature to use unsplash as a resource for pictures for a day, watching tutorials, reading medium articles and stack overflow threads. It would just not come together. Before it got to a level of frustration that makes it hard to get back from, I took a break, went for a run, had something to eat and listened to my audiobook for 5 minutes while my mind was already heading back to coding. And low and behold, I read one more medium article, and all of a sudden it did work! All of the stress and insecurities and frustration just went away and were forgotten about and I moved on to the next feature - after staging my stages to git of course!

    The process taught me the value of stepping back and approaching a problem with a fresh perspective.

  • Always have your server logs open when coding

    When creating a new card, the user can pick more than one deck in which the card should be placed. However this did not work. After studying the server logs in real time for a few tries, and using tools such as Postman and the useful console.log(something I want to know, such as data), it turned out that the data set that was passed along to the database had a slightly different syntax than expected. Namely the array of deck-ids were not found within the card parameters.

              
           Parameters: {“word"=>"nuevo", "translation"=>"new", "picture"=>"", "hint"=>"", "flipped"=>false, "deck_ids"=>[12, 11], "card"=>{"word"=>"nuevo", "translation"=>"new", "picture"=>"", "hint"=>"", "flipped"=>false}}

    It turns out that I had to change the axios syntax in the store very slightly from

    return axios.post('/cards', card)

    to

    return axios.post('/cards', {card: card}). 
              
             Parameters: {“card"=>{"word"=>"el maestro", “translation"=>"the teacher", "picture"=>"", "hint"=>"", "flipped"=>false, "deck_ids"=>[12, 11]}}

    Credit for finding and fixing the error goes to Dimi. Some things just come with experience, I would not have found the solution to that online, and it is still unclear, why only deck-ids was placed outside of the card parameters.

  • Create a document to monitor my workflow and productivity
    • All tasks go into a spreadsheet, with an ID and overview when I started and finished the task
    • A week to week overview with the tasks I want to tackle: a list above and then assigning them to each day, dividing it into “Caculation" and “Realtime"
    • Besides managing my time better and not drop the ball on anything, it also helps to see how long tasks actually take. This will in turn make planning for new or similar tasks and projects more realistic.
  • Issue Tracker
    • Stephanie and I started using GitLab's issues to track discussions and document our decision making process and results. We are only a team of two at the moment, but for future projects we want to collaborate and work in larger teams, that will most likely work in an asynchronous manner.
    • By opening issues and linking/assigning the members that should be involved in the decision making process and/or will be affected by the results, everyone is up to date on the status, can ask questions to clarify points, and contribute to the final results, independently from other team members making time to bring them up to speed.
    • The Issue opener can serve as a moderator for the issue or assign someone else to take charge.
    • Nothing gets dropped, the issues can serve as a to do list, it is easily visible if issue still needs resolving or are closed.
    • Issues document when decisions were made, why, how and by whom. Others can reread the documentation in an efficient way, instead of having to set up a chat with coworkers - who might work in different time zones or don’t have room in their schedule in a timely manner.
    • All discussions are linked to the project, all essential information is gathered in one place and available for other collaborators or new team members. They don't have to give them access to googledDocs or sending phone protocols back and forth.
    • There are other options such as Milestone or Due Date, that we will still have to include in the issues.
  • Starting an Error Log
    • I started a file to record the errors I encounter along the way, plus the solution. This way with a reoccurring error I know where to check first, because contrary to my own believe, I will surely remember next time how I solves this” is not always true. unfortunately; Especially helpful when start using a new programming language or framework, and you are still not familiar with all the errors.
